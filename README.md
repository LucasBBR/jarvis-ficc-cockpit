# Jarvis FICC Cockpit

Static front-end prototype for a BTG Pactual-style FICC client cockpit. The current experience is a client-side demo focused on interaction design, state handling, motion, notes, book views, document/credit KPIs, and a Jarvis chat/insights surface. There is no backend yet.

Live site: https://lucasbbr.github.io/jarvis-ficc-cockpit/

## What This Is

- A single-page static web app rendered by plain JavaScript.
- Mock data for the Vale economic group and FICC workflows.
- A front-end sandbox for future backend integration.
- A GitHub Pages deployment from the `main` branch.

## Project Structure

```text
.
├── index.html                  # Primary deployed entrypoint
├── Jarvis FICC Cockpit.html    # Local alias of the same entrypoint
├── data.js                     # Mock data exposed as window.JV_DATA
├── cockpit.js                  # Main renderer, state machine, event handlers
├── app.css                     # Component/layout styles and interaction polish
├── ds/
│   ├── colors_and_type.css     # Design-system tokens, fonts, semantic colors
│   └── fonts/README.txt        # Font substitution notes
├── .nojekyll                   # Required so GitHub Pages serves ds/ paths as-is
└── .gitignore                  # Ignores screenshots, uploads, JSX scratch files
```

Some local `.jsx` files may exist from earlier React explorations. They are ignored by git and are not loaded by `index.html`. The deployed app uses only `data.js`, `cockpit.js`, `app.css`, and `ds/colors_and_type.css`.

## Runtime Flow

1. `index.html` loads design tokens from `ds/colors_and_type.css`.
2. `index.html` loads app styles from `app.css`.
3. `data.js` creates `window.JV_DATA`.
4. `cockpit.js` reads `window.JV_DATA`, owns all UI state, renders into `#root`, and binds delegated event handlers.

The current renderer is intentionally framework-free. Most UI is generated by `render*` functions in `cockpit.js`; CSS classes in `app.css` provide the visual system.

## Key UI Areas

- **Top bar KPIs:** Coverage, P&L, Documents, Credit, and Hit Ratio popovers.
- **Notes:** Personal/team/all note filters, rich text, hashtag suggestion/pill behavior, voice demo, save states.
- **Book:** Trades/positions by product and member filter.
- **Documents:** CGD/FX tabs, counterparty filter, status badges.
- **Credit:** Approved/used/available summary, counterparty pagination/filter.
- **Jarvis:** Floating entrypoint, chat drawer, collapsible insights rail, like/dislike feedback, thinking animation.
- **Demo controls:** Hidden behind the `Demo` pill and `Shift+D`.

## Run Locally

No install step is required.

Option 1: open the file directly.

```powershell
Start-Process .\index.html
```

Option 2: serve the folder locally.

```powershell
python -m http.server 8000
```

Then open:

```text
http://localhost:8000/
```


```powershell

```

## Design Rules

- Keep the app enterprise/minimal: dense but readable, no decorative motion.
- Reuse existing tokens from `ds/colors_and_type.css`.
- Preserve the sticky top bar and KPI strip behavior.
- Avoid making color the only status signal; pair status with text/icons.
- Respect reduced motion via the existing `state.reduceMotion` preference.
- Keep side/action colors neutral unless they represent financial value movement or explicit status.

## Backend Integration Notes

The future backend should replace mock data and simulated timers without changing the high-level UI contract:

- Replace `window.JV_DATA` with hydrated data from an API or bootstrapped payload.
- Keep panel-level states: empty, loading, refreshing, stale, degraded, error, success.
- Preserve small, explicit status tags for stale/disconnected data.
- Route Jarvis chat and insights through backend endpoints while keeping client-side feedback state immediate.
